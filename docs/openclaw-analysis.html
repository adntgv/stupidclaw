2. Classification: Reuse vs Borrow vs Rebuild

SubsystemVerdictReasoning
Gateway + Message Router CAN REUSE Model-agnostic WebSocket router. StupidClaw could connect as a backend agent via WS. The gateway doesn't care what model processes the message.
Channel Adapters CAN REUSE Telegram/Discord/WhatsApp adapters are independent of the agent. If StupidClaw uses OpenClaw's gateway, it gets all channels for free.
Cron/Heartbeat System CAN REUSE Scheduling is model-independent. Cron triggers messages into sessions — doesn't matter if the session runs Opus or Llama 8B.
Config System CAN REUSE Schema validation, hot reload, env vars. Pure infrastructure.
Security (audit, SSRF, redact) BORROW Good patterns: SSRF protection, secret redaction, audit logs. Reimplement in Python — the logic is universal but the Node.js code can't be imported.
Skill System BORROW SKILL.md convention is excellent. Copy the file format and loading pattern. But skill injection into prompts needs rethinking — small models can't afford 2K tokens of skill instructions.
Memory Convention BORROW MEMORY.md, SOUL.md, USER.md, daily files — these are just markdown files. Copy the convention exactly. But how they're injected into context must change.
Session Management BORROW Session keys, routing logic, multi-user support. Good design, reimplement in Python.
Tool Definitions BORROW Tool metadata format works. But execution needs stricter validation for small models.
Agent Loop REBUILD THE critical difference. OpenClaw sends everything to a smart model and trusts it. StupidClaw needs a rigid state machine: classify → plan → execute → verify → respond.
Context Building REBUILD OpenClaw dumps SOUL.md + USER.md + AGENTS.md + MEMORY.md + full history + tools = 50K+ tokens. StupidClaw has 8K max. Must aggressively compress, summarize, and prioritize.
Tool Calling REBUILD OpenClaw trusts the model to generate correct tool JSON. Small models produce malformed JSON ~20% of the time. Need: constrained output, schema validation, retry logic, fallback.
Sub-agent Orchestration REBUILD OpenClaw lets the model decide when to spawn. Small models can't handle meta-cognition. Need explicit rules: "if task has >3 steps AND estimated time >30s → decompose."
Self-Modification REBUILD OpenClaw freely edits AGENTS.md, SOUL.md, HEARTBEAT.md. Small models can't be trusted to rewrite their own instructions. Need: templates, diff-based edits, human approval.
MCP Integration REBUILD MCP requires reliable tool-use and long context. Small models struggle with complex tool schemas. Need simplified tool wrappers around MCP servers.

3. Why The Agent Loop Must Be Rebuilt

The Core Problem: OpenClaw's agent works because GPT-4/Claude can handle a 100K token prompt with 50 tools and figure out what to do. A Llama 8B model with 8K context simply cannot.

OpenClaw's Approach (Smart Model)

User Message
    ↓
[Inject: SOUL.md + USER.md + AGENTS.md + MEMORY.md + 
 HEARTBEAT.md + conversation history + all available 
 tools + skill descriptions + workspace files]
    ↓
Send 50K+ tokens to Claude Opus / GPT-4
    ↓
Model decides: answer directly OR call tools OR spawn sub-agent
    ↓
If tools: execute → feed result back → model continues
    ↓
Response (model handles formatting, tone, everything)

StupidClaw's Approach (Dumb Model)

User Message
    ↓
CLASSIFY (tiny prompt, 200 tokens)
→ Is this: question | task | chitchat | unclear?
    ↓
PACK CONTEXT (code, not model)
→ Select ONLY relevant memories (semantic search)
→ Inject ONLY needed tool descriptions
→ Summarize history to 500 tokens max
    ↓
PLAN (structured prompt, 1K tokens)
→ "Break this into max 3 steps. Output JSON."
→ Schema-validated. Retry if malformed.
    ↓
EXECUTE (code, not model)
→ Run tools deterministically
→ Collect results
    ↓
VERIFY (tiny prompt, 500 tokens)
→ "Did the tool results answer the question? yes/no"
→ If no: retry or escalate
    ↓
RESPOND (structured prompt, 1K tokens)
→ "Given these results, answer the user. Be brief."
    ↓
Response
Key Insight: OpenClaw makes 1 smart LLM call per turn. StupidClaw makes 3-5 dumb LLM calls per turn — but each call is simple, constrained, and verifiable. Total tokens used is often LESS than OpenClaw's single massive prompt.

4. The Hybrid Architecture Option

Best of Both Worlds: Instead of replacing OpenClaw, StupidClaw could plug INTO OpenClaw's infrastructure.

Option A: Full Replacement (Current Approach)

[Telegram] → [StupidClaw Bot] → [StupidClaw Agent] → [Groq API]
                                       ↓
                              [Tools, Memory, etc.]

Pros: Simple, independent, no Node.js dependency
Cons: Must reimplement channels, cron, config, sessions from scratch

Option B: OpenClaw Gateway + StupidClaw Agent Backend

[Telegram/Discord/WhatsApp] → [OpenClaw Gateway] → [StupidClaw Agent via WS]
                                     ↓                        ↓
                              [Cron, Config,          [Rigid State Machine]
                               Sessions]               [Groq/Cerebras API]
                                                       [Tools, Memory]

Pros: Free channels, cron, config, sessions. Focus on the agent brain only.
Cons: Dependency on OpenClaw. Must understand WS protocol. Node.js runtime required.

Option C: Standalone Now, Gateway Later

Phase 1: [Telegram] → [StupidClaw standalone] → [Groq]
Phase 2: Add OpenClaw Gateway as optional infrastructure layer

Pros: Ship fast, add complexity later
Cons: Some rework when integrating gateway

Recommendation: Option C. Build standalone first (it's working already). Design the agent interface so it can later accept messages from OpenClaw's gateway via WebSocket. This keeps velocity high while preserving the option to scale.

5. What Makes Extensibility Hard for Small Models

OpenClaw CapabilityWhy It Works with Smart ModelsWhy It Breaks with Small ModelsStupidClaw Solution
Self-modification Claude can read AGENTS.md, understand what to change, make precise edits 8B model will corrupt files, miss context, make nonsensical edits Template-based edits. Model fills in blanks, code applies changes. Human approval for risky edits.
Skill selection Model reads 50 skill descriptions and picks the right one Can't fit 50 skill descriptions in 8K context Embedding-based skill matching. Code selects top-2 skills by semantic similarity. Model only sees relevant ones.
Sub-agent spawning Model decides "this task needs a sub-agent" based on judgment Small models can't judge task complexity reliably Rule-based decomposition: if plan has >3 steps or estimated duration >60s → auto-decompose into sub-tasks.
MCP tool use Model understands complex tool schemas with nested parameters Small models fail on complex JSON schemas Simplified wrapper tools. MCP server exposed as simple functions: plane_create_issue(title, description) instead of raw MCP protocol.
Multi-turn reasoning Model maintains coherence across 20+ turns with full history Context window too small for long conversations Aggressive summarization. Every 5 turns → summarize → replace history. Semantic recall for older context.
Error recovery Model notices errors in tool output and self-corrects Small models often don't notice errors, or make things worse Explicit verification step. Separate LLM call: "Did this tool output match the expected result? yes/no/error"

6. Minimal Platform Layer for Extensibility

To make StupidClaw a platform (not just a bot), it needs these core abstractions:

6.1 Tool Registry

# tools/registry.py
class ToolRegistry:
    tools: dict[str, Tool]  # name → Tool
    
    def register(self, tool: Tool)
    def get(self, name: str) → Tool
    def list_for_context(self, max_tokens: int) → str
    def match_by_intent(self, query: str) → list[Tool]  # semantic matching

class Tool:
    name: str
    description: str       # short, for context injection
    parameters: dict       # JSON schema
    risk_level: str        # low | medium | high
    requires_confirmation: bool
    execute(args) → str

6.2 Skill System

# skills/loader.py
class SkillLoader:
    def scan(self, skills_dir: str) → list[Skill]
    def match(self, query: str) → Skill | None  # embedding-based
    def load_into_context(self, skill: Skill, budget: int) → str

class Skill:
    name: str
    trigger_patterns: list[str]  # "when to use"
    tools_required: list[str]
    prompt_template: str
    max_tokens: int  # context budget this skill needs

6.3 Memory Interface

# memory/interface.py
class Memory:
    hot: HotMemory           # last N messages
    semantic: SemanticMemory  # ChromaDB vectors
    episodic: EpisodicMemory  # daily JSON logs
    procedural: dict          # MEMORY.md parsed

    def recall(self, query: str, budget: int) → str
    def store(self, fact: str, source: str)
    def summarize_session(self) → str

6.4 Agent Interface (Pluggable)

# agent/interface.py
class AgentBackend:
    """Any agent implementation must satisfy this interface."""
    
    def process(self, message: UserMessage, session: Session) → AgentResponse
    
class UserMessage:
    text: str
    sender_id: str
    channel: str  # telegram | discord | web
    attachments: list

class AgentResponse:
    text: str
    tools_used: list[ToolCall]
    confidence: float
    tokens_used: int
    cost: float
With these 4 abstractions, StupidClaw becomes a platform:
  • Add new tools by implementing Tool and registering
  • Add new skills by dropping SKILL.md files
  • Swap memory backends (ChromaDB → pgvector → FAISS)
  • Swap agent implementations (rigid FSM → ReAct → future approaches)
  • Connect to OpenClaw gateway later via the AgentBackend interface

8. Conclusion

StupidClaw is not a fork of OpenClaw. It's a different philosophy that borrows OpenClaw's shell.

Reuse: File conventions (MEMORY.md, SOUL.md), gateway infrastructure (later), cron scheduling
Borrow: Skill system design, tool metadata format, session management patterns
Rebuild: Agent loop (rigid FSM), context packing (aggressive compression), tool calling (constrained output), self-modification (templated), sub-agents (rule-based)

The fundamental insight: OpenClaw puts intelligence in the model. StupidClaw puts intelligence in the system.
