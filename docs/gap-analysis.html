3. Memory Architecture

Small models (4K-8K context) need aggressive memory management. Here's a 4-tier system that's cheap and effective:

Tier 1: Hot Session Memory (In-Memory)

What: Current conversation context (last 5-10 messages)

Storage: Python list/dict in RAM

Token Budget: ~2K tokens max

Strategy: Sliding window. When context fills, summarize oldest messages and move to Tier 2.

Implementation:

class HotMemory:
    def __init__(self, max_tokens=2000):
        self.messages = [] # [(role, content, timestamp)]
        self.max_tokens = max_tokens
    def add(self, role, content):
        # Add message, trim oldest if over budget
        # Call summarize() on trimmed messages → move to Tier 2

Tier 2: Semantic Memory (Vector DB)

What: Searchable long-term memory of conversations, facts, user preferences

Storage: ChromaDB (SQLite-based, zero-cost, local)

Embeddings: Free options:

  • sentence-transformers/all-MiniLM-L6-v2 (384-dim, fast, local)
  • Voyage AI free tier (1M tokens/month)
  • Groq + llama-guard-3-8b (text embeddings via hidden states)

Strategy: Index every message chunk. On new query, semantic search retrieves top 3-5 relevant memories.

Query Pattern:

# User asks: "What did we discuss about Python last week?"
relevant = semantic_search("Python discussion last week", limit=5)
# Inject into context: "Relevant memories: {relevant}"

Tier 3: Episodic Memory (Structured Logs)

What: Daily logs, task completions, events (like OpenClaw's daily .md files)

Storage: JSON or SQLite (e.g., memory/YYYY-MM-DD.json)

Schema:

{
  "date": "2026-03-06",
  "events": [
    {"time": "14:30", "type": "task_completed", "task": "Draft email", "outcome": "success"},
    {"time": "15:45", "type": "search", "query": "Python asyncio", "results_count": 8}
  ]
}

Strategy: Log every action. Retrieve recent logs (last 7 days) on session start. Search by date/type/keyword.

Tier 4: Procedural Memory (Learned Patterns)

What: Reusable snippets, shortcuts, user preferences (like OpenClaw's MEMORY.md)

Storage: Single MEMORY.md or memory.json file

Content Examples:

  • User prefers Python over JavaScript
  • User's timezone: GMT+5
  • Common commands: git commit -am
  • Lessons learned: "Always validate API responses before parsing"

Strategy: Always inject into system prompt. Manually curated (agent suggests updates, user approves).

4. Context Packing Strategy

The Small Context Window Problem: Groq llama-4-scout-17b has ~8K tokens. After system prompt (~1K) and response budget (~1K), you have only ~6K for conversation history, memory, tool outputs, and instructions.

Smart Context Budget Allocation

Component Token Budget Strategy
System Prompt 600-800 Keep concise. Define agent role, tool usage rules, response format.
Procedural Memory 300-500 MEMORY.md equivalent. User prefs, shortcuts, key facts. Always included.
Semantic Retrieval 500-800 Top 3-5 relevant past messages. Query-based. Optional.
Recent Messages 2000-3000 Last 5-10 turns. Sliding window with summarization.
Tool Outputs 1000-1500 Truncate long outputs. Summarize web pages. Compress JSON.
Response Budget 800-1200 Reserve for agent's reasoning + response.

Context Compression Techniques

1. Message Summarization (Aggressive):

When conversation exceeds budget, summarize oldest messages:

# Before: 10 messages × 200 tokens = 2000 tokens
# After: 1 summary × 150 tokens + 5 recent messages × 200 = 1150 tokens

Free summarization: Groq llama-3.1-8b (fast, free) or local Phi-3.5-mini (3.8B, CPU-friendly)

2. Semantic Chunking:

Split long messages into topic-based chunks. Store in vector DB. Retrieve only relevant chunks.

# Instead of: "Here's a 2000-token web page..."
# Store: 10 chunks × 200 tokens, retrieve top 3 relevant chunks = 600 tokens
3. Tool Output Truncation:

Web searches return 10 results? Truncate to top 3. Code output 500 lines? Show first 50 + last 50.

def truncate_output(text, max_tokens=500):
    if count_tokens(text) > max_tokens:
        return text[:max_tokens*4] + "\n... [truncated]"
    return text
4. Priority-Based Loading:

Not every message is equally important. Prioritize:

  1. Current user message (highest priority)
  2. Last agent response (for continuity)
  3. Tool outputs (if referenced in current message)
  4. Semantic memories (query-relevant)
  5. Recent messages (fill remaining space)

import requests

def brave_search(query, count=5):
    url = "https://api.search.brave.com/res/v1/web/search"
    headers = {"X-Subscription-Token": BRAVE_API_KEY}
    params = {"q": query, "count": count}
    response = requests.get(url, headers=headers, params=params)
    results = response.json()["web"]["results"]
    return [{"title": r["title"], "url": r["url"], "snippet": r["description"]} for r in results]

Recommended: ReAct Pattern (Simple Implementation)

The ReAct Pattern: Reasoning (think) + Acting (use tools) in a loop.

# Pseudo-code for ReAct agent tools = { "web_search": web_search_function, "read_file": read_file_function, "write_file": write_file_function, "shell": shell_exec_function } def react_agent(user_query, max_steps=5): context = [{"role": "system", "content": SYSTEM_PROMPT}] context.append({"role": "user", "content": user_query}) for step in range(max_steps): # Step 1: Reasoning (LLM thinks) response = llm.generate(context) # Step 2: Parse action action = parse_action(response) # Extract: {"tool": "web_search", "args": {"query": "..."}} if action["tool"] == "final_answer": return action["args"]["answer"] # Step 3: Execute tool if action["tool"] in tools: result = tools[action["tool"]](**action["args"]) context.append({"role": "assistant", "content": response}) context.append({"role": "user", "content": f"Observation: {result}"}) else: context.append({"role": "user", "content": "Error: Unknown tool"}) return "Max steps reached without answer."
Why ReAct Works for Small Models:
  • Simple loop: No complex planning, just "think → act → observe" repeatedly
  • Explicit reasoning: Forces model to explain its thinking before acting (reduces errors)
  • Proven: Research shows ReAct outperforms chain-of-thought for tasks requiring external tools
  • Few-shot learnable: Small models can learn ReAct from 2-3 examples

7. Verification & Grounding

The Hallucination Problem: Small models like llama-4-scout-17b hallucinate more than GPT-4. You MUST implement verification layers.

Grounding Strategies

1. Tool Output Verification (Automatic):

Always show tool outputs to the model. Force it to ground responses in observations.

# BAD: Let model answer without grounding
"What's the weather?" → Model hallucinates "It's sunny"

# GOOD: Ground in tool output
"What's the weather?" → web_search("weather") → "Observation: Rainy, 15°C" → Model: "It's rainy and 15°C"
2. Constrained Output Formats (Force Structure):

Don't let model free-form respond. Force JSON/structured output.

# System prompt:
"You must respond in JSON format: {"reasoning": "...", "action": {"tool": "...", "args": {...}}}"

# This prevents rambling and hallucination
3. Fact-Checking via Search (For Critical Claims):

When model makes a factual claim, automatically verify via web search.

# Model says: "Python 3.12 was released in October 2023"
# Auto-verify: web_search("Python 3.12 release date")
# If mismatch: Flag to user OR force model to correct
4. Confidence Scores (Explicit Uncertainty):

Force model to rate confidence (1-5) on each response. Warn user if confidence < 3.

# Output format:
{"answer": "...", "confidence": 4, "sources": ["url1", "url2"]}
5. Human-in-the-Loop (For High-Stakes Actions):

Require confirmation before destructive actions (file deletion, API calls, purchases).

# Agent wants to: delete_file("important.txt")
# System: "Confirm? [y/n]"
# Only execute if user approves

Hallucination Mitigation Checklist

Structured outputs (JSON) High Low (prompt engineering) Always
Fact-checking via search Medium Medium (auto-search hook) Critical facts only
Confidence scores Medium Low (add to output schema) User-facing responses
Human-in-the-loop High Medium (confirmation UI) Destructive actions

Appendix: Key Learnings from Research

What Makes Agents Work with Small Models?
  1. Code > JSON: Agents that write actions as Python code outperform JSON-based tool calling by ~30% (smolagents research)
  2. Few-shot is enough: Small models learn tool usage from 2-3 examples (no fine-tuning needed)
  3. Grounding is critical: Always show tool outputs. Models hallucinate less when forced to ground in observations
  4. Simplicity wins: ReAct pattern (think → act → observe) beats complex planning for <70B models
  5. Memory architecture matters: 4-tier memory (hot + semantic + episodic + procedural) fits small contexts
  6. Free tools exist: Brave API (2K/mo), ChromaDB (local), SearXNG (self-hosted) = $0 operating cost
  7. Context is precious: With 8K tokens, every token counts. Aggressive compression + priority loading is mandatory
